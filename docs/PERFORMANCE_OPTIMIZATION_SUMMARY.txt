BOTTLENECK ANALYSIS RESULTS
============================

Bottleneck #1: RPC Semaphore (RESOLVED ✅)
------------------------------------------
Symptom: Upload stuck at 2.4 ops/sec regardless of workers
Root Cause: 64 concurrent RPC slots vs 600 needed (50 workers × 12 chunks)
Solution: Increased to 512 concurrent RPC calls
Impact: +6,030% throughput improvement
Files: src/net/rpc.c, src/storage/parallel_chunks.c

Bottleneck #2: libuv Thread Pool (TESTED - NOT THE ISSUE)
----------------------------------------------------------
Hypothesis: Default 4 threads limiting I/O parallelism
Test: Increased UV_THREADPOOL_SIZE to 64
Result: Only +3.3% improvement
Conclusion: Not the bottleneck

Bottleneck #3: Physical Disk I/O (CURRENT CEILING 🔴)
-----------------------------------------------------
Evidence:
  - CPU Usage: 0% (not CPU-bound)
  - Load Average: 31.78 (threads waiting on I/O)
  - 24 virtual disks → 1 physical disk
  - ~1,800 writes/sec hitting fsync() saturation

Root Cause:
  - Single physical disk handling all I/O
  - fsync() calls serialize on disk queue
  - Disk capacity: ~200-500 fsyncs/sec max
  - System demand: ~1,800 writes/sec

Current Ceiling: ~150 ops/sec (37.7 MB/s)

NEXT OPTIMIZATION PATHS
========================

Option 1: fsync Optimization (CODE - MODERATE)
----------------------------------------------
Techniques:
  - Batch multiple chunks before fsync
  - Use fdatasync() instead of fsync() (skip metadata)
  - Write-behind caching with periodic flush
  - Group barrier (collect N writes, fsync once)

Expected: 3-5x improvement (450-750 ops/sec)
Trade-off: More complex error handling
Effort: 2-3 days development + testing

Option 2: Real Distributed Hardware (INFRASTRUCTURE)
----------------------------------------------------
Setup:
  - 6 physical machines (or VMs with dedicated disks)
  - Each node on separate storage device
  - Network becomes new bottleneck variable

Expected: 10-20x improvement (1,500-3,000 ops/sec)
Reality: Current localhost testing is artificial constraint
Effort: Infrastructure provisioning

Option 3: Async I/O (CODE - HIGH COMPLEXITY)
--------------------------------------------
Approach:
  - Use io_uring (Linux 5.1+) or libaio
  - Avoid blocking on fsync
  - Pipeline writes more efficiently

Expected: 2-3x improvement
Challenge: Complex implementation, error handling
Effort: 1-2 weeks development + testing
